# Automated dependency updates via the hosted Renovate GitHub App

`.github/renovate.json` configures Renovate to open PRs for npm `devDependencies` and pinned GitHub Actions on a weekly schedule, running as the hosted GitHub App rather than a self-hosted workflow. We chose the App because it authenticates as a proper GitHub App installation rather than the default `GITHUB_TOKEN`, so its PRs trigger `pull-request.yml` (including commitlint) normally. A self-hosted workflow authenticating with `GITHUB_TOKEN` would not get that for free: GitHub deliberately withholds `pull_request` triggers from PRs opened by the default token, and working around it would need a fine-grained PAT or a dedicated internal GitHub App, extra infrastructure this repo doesn't otherwise need.

Renovate's `minimumReleaseAge` is set to `3 days`, mirroring the `minimumReleaseAge: 4320` / `trustPolicy: no-downgrade` already enforced at install time in `pnpm-workspace.yaml`. Without this, Renovate could open a PR for a package pnpm itself would then refuse to install as too new. This gate applies uniformly, including to `vulnerabilityAlerts`: a disclosed CVE with a same-day fix still waits out the 3-day window like any other update, rather than bypassing it. This was a deliberate choice, not an oversight. The alternative, bypassing the age gate for security fixes, closes the "known CVE" exposure faster but reopens exactly the "trust a brand-new, unvetted release" risk the gate exists to prevent, on the one class of dependency change most likely to be a supply-chain attack vector.

`peerDependencies` (`typescript`, `@types/node`) are excluded from Renovate entirely. Bumping a peer range changes what TypeScript/Node versions this package promises to support, which is the same kind of deliberate, manual decision [ADR 0003](0003-manual-version-bumping.md) already established for this package's own version, not something that should happen because a bot merged a PR.

## Considered options

- Self-hosted Renovate via a GitHub Actions workflow, matching the repo's existing `dnd-mapp/action-*` pattern. Rejected: solving the `GITHUB_TOKEN` PR-trigger gap requires a PAT or a dedicated internal GitHub App, with no benefit over the hosted App for a repo that has no need to reach a fully private, token-inaccessible registry.
- Bypassing `minimumReleaseAge` for `vulnerabilityAlerts` specifically, to close disclosed CVEs same-day. Rejected in favor of a consistent stability window across all update types, including security fixes; manual review of the resulting PR is treated as the trust signal instead of elapsed time.
