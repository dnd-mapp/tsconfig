# Package is published privately to GitHub Package Registry instead of the public npm registry

`release.yml` now publishes `@dnd-mapp/tsconfig` to GitHub Package Registry (`npm.pkg.github.com`), scoped private to the `dnd-mapp` org, instead of the public npm registry. This package was never actually consumed by anything (its one published npm version, `1.0.0`, has been unpublished), so there was no external audience to preserve. The org wants this package to stay internal, and GitHub Package Registry gives private, org-scoped distribution for free, which the public npm registry does not (private packages there require a paid npm org).

This supersedes [ADR 0002](0002-staged-and-trusted-publishing.md). Npm's OIDC trusted publishing and staged 2FA-approval flow are both npm-registry-specific; GitHub Package Registry has no equivalent to either. Publishing now authenticates with the workflow-scoped `GITHUB_TOKEN` (`packages: write`), and there is no post-publish staging/approval step: the moment the `release` job's publish step runs, the version is live.

That also changes the release approval story from [ADR 0004](0004-release-workflow-job-structure.md). ADR 0004 reasoned about two independent human-approval gates: the `npm-publish` GitHub Environment before the job starts, and npm's own staged 2FA approval after CI submits the package. With the second gate gone, the single environment approval gate (renamed `package-publish`) is now the only point at which a human confirms a release before it goes live. This is an accepted reduction, not an oversight: ADR 0002's two-gate design existed to compensate for OIDC trusted publishing removing the human-presence guarantee on a *public* registry with no other access control. A private, org-scoped package doesn't carry that same anonymous-public-consumer risk, so the second gate no longer pays for itself.

The removal of the second gate also removes the reason `release.yml` created the GitHub release as a draft. ADR 0004 created it as a draft specifically so it could stay unpublished until npm's staged approval also went through, keeping the two "this is out" signals (GitHub release, npm registry) in sync. With no second signal to wait on, the release is now created and made visible in the same step as the package publish, immediately after the single environment approval.

One consequence has no automated fix: GitHub Package Registry has no API/CLI flag to preset a package as private before it exists, and no `publishConfig.access`-equivalent field. The first publish creates the package, and it defaults to public visibility inherited from this (public) repository until an org admin manually flips it to private in the package's GitHub settings. Until that manual step happens, a version could technically be visible to more than the org. This is accepted as a one-time, human-performed step rather than something the workflow can guarantee.

## Considered options

- Keep publishing to the public npm registry, using a paid npm org to get private package support. Rejected: adds a paid, separate account/billing relationship for no benefit over GitHub Package Registry, which the org already has access to via its existing GitHub org.
- Preserve a two-gate approval model by adding a second environment/manual step after publish (since GitHub Package Registry has no native staging feature to lean on). Rejected: the threat model that motivated the second gate in ADR 0002 (anonymous public consumers of a compromised publish) doesn't apply to a private, org-scoped package; a second gate here would just be process for its own sake.
- Dual-publish to both the public npm registry and GitHub Package Registry. Rejected: the package isn't consumed anywhere yet, and the org wants it private going forward, so there's nothing dual-publishing would preserve.
